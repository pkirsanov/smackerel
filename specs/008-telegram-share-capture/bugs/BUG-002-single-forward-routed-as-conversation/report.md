## Adversarial Property

Pre-fix, `flushConversation` had no `len(buf.Messages) == 1` branch. The new test `TestBUG002_SC_TSC09_SingleForward_NotConversation` inspects the raw JSON body posted to the capture endpoint and asserts the body has NO `conversation` key. Reverting the fix (removing the `if len(buf.Messages) == 1 { return b.flushSingleForward(ctx, buf) }` line) would cause the test to fail because the body would contain a `conversation` block. The URL-case test enforces the same property for the dead-code URL-detection branch in `captureSingleForward`. The strengthened assembler test enforces the property at the assembler level.

The over-correction guard (`TestBUG002_TwoForwards_StillProduceConversation`) ensures the fix cannot be naively widened to bypass `conversation` for all forwards — three messages must still emit a `conversation` body with `message_count == 3`.

### Summary

BUG-002 was filed 2026-04-26 by `bubbles.workflow` (bugfix-fastlane). The wired
`Bot` always constructs a `ConversationAssembler`, so
`handleForwardedMessage` in [internal/telegram/forward.go](../../../../internal/telegram/forward.go)
routed every forwarded message — including a single forward — through
`b.assembler.Add(...)`. When the assembler's inactivity timer expired,
`flushConversation` in [internal/telegram/bot.go](../../../../internal/telegram/bot.go)
emitted a `conversation`-typed capture body even when `len(buf.Messages) == 1`,
violating [SC-TSC09](../../spec.md) and rendering the URL-detection branch in
`captureSingleForward` dead code on the wired path.

The fix added a 1-message short-circuit at the top of `flushConversation` that
delegates to a new `flushSingleForward` helper in
[internal/telegram/forward.go](../../../../internal/telegram/forward.go). The helper
reconstructs `ForwardedMeta` from the buffer and applies the same URL-detection /
text-capture logic as `captureSingleForward`, preserving SC-TSC09 semantics and the
URL artifact path. Multi-message conversation behavior (SC-TSC08) is unchanged, the
assembler API is unchanged, and there are no schema or config changes.
