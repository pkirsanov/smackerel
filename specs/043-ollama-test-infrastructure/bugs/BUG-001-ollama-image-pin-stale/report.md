# BUG-001 Report

> Generated by `bubbles.bug` (bugfix-fastlane), 2026-05-10.

## Summary

Spec 043 Scope 01 pinned `ollama/ollama:0.6` for both `infrastructure.ollama.image` and `infrastructure.ollama.test.image`. That tag was yanked from Docker Hub. Any environment that auto-starts Ollama (the test env auto-starts via `environments.test.ollama_enabled: true` per spec 043 Scope 03) failed to start because the image manifest now returns 404. The fix re-pins to `ollama/ollama:0.23.2` (latest stable per `github.com/ollama/ollama/releases/latest`, verified live HTTP 200 against the Docker Hub Tags API at fix time) and adds an adversarial registry-existence guard so future yanked tags fail loudly inside an integration run instead of inside `docker compose up`.

## Adversarial Property

Pre-fix, `config/generated/test.env` carried `OLLAMA_IMAGE=ollama/ollama:0.6`. The new test file's `dockerHubTagExists` helper, applied to that same input, returns HTTP 404 (proven by `TestOllamaImagePinIsPublished_AdversarialYankedTag`). Therefore, had the new test existed at the time spec 043 Scope 01 landed, `TestOllamaImagePinIsPublished_LiveTag` would have failed loudly with the same 404 the adversarial sub-case demonstrates. The two tests share a single helper, so the adversarial proves the live test is not tautological — different inputs route through identical code and produce different outputs depending on real Docker Hub registry state.

If a future commit regresses `OLLAMA_IMAGE` to any yanked tag (whether `0.6` or some future yank), the live test will fail with HTTP 4xx inside the integration run, before the test stack tries to pull the image. This converts a `docker compose up` failure deep inside the test orchestration loop into a fast, named test failure with a clear actionable error message.
