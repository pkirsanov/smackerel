# BUG-002 Report

> Generated by `bubbles.bug` (bugfix-fastlane), 2026-05-10. Populated as fix lands and gates run.

## Summary

BUG-001 bumped the ollama image from `ollama/ollama:0.6` → `ollama/ollama:0.23.2`. The new image ships only the `ollama` binary in PATH — it does NOT ship `wget` or `curl`. The compose-side healthcheck in `docker-compose.yml` line 182 (and the deploy-side mirror `deploy/compose.deploy.yml` line 199) was hardcoded to `["CMD-SHELL", "wget … http://localhost:${OLLAMA_CONTAINER_PORT}/api/tags || exit 1"]`. Every healthcheck invocation in the test stack exited 127 (`executable file not found in $PATH`), so `docker compose up -d --wait --wait-timeout 80` exited 124 with `container smackerel-test-ollama-1 is unhealthy`. This blocked `./smackerel.sh test integration` at the stack-up step — every downstream Go integration test failed to start. Spec 044 validate phase was hard-blocked.

The fix replaces the `wget`-based healthcheck with `["CMD", "ollama", "list"]` in both compose files. `ollama list` is in the image at `/usr/bin/ollama`, on PATH, and is a real liveness signal — it returns exit 0 only when the daemon is reachable on `OLLAMA_HOST` (defaults to `http://127.0.0.1:11434` inside the container, matching `OLLAMA_CONTAINER_PORT=11434`) and exit 1 otherwise. A new adversarial integration test (`tests/integration/ollama_healthcheck_test.go`) parses both compose files and asserts the ollama healthcheck's first command token is in an in-image-binary allowlist; three adversarial sub-cases prove the validator rejects `wget`/`curl` (CMD form) and `wget` (CMD-SHELL form) with explicit error messages naming the offending binary AND the image.

## Adversarial Property

Pre-fix, `docker-compose.yml` services.ollama.healthcheck.test was `["CMD-SHELL", "wget … || exit 1"]`. The new test file's `assertOllamaHealthcheckUsesInImageBinary` helper, applied to that same input, returns a non-nil error mentioning `wget` and `ollama/ollama:0.23.2` (proven by `TestOllamaHealthcheck_AdversarialCMDShellWrappedWget`). Therefore, had the new test existed at the time the original spec 043 healthcheck was authored OR at the time BUG-001 bumped the image, `TestOllamaHealthcheck_LiveFiles` would have failed loudly with a clear message explaining the missing binary. The four tests share a single helper, so the three adversarial sub-cases prove the live test is non-tautological — different inputs route through identical code and produce different outputs depending on whether the first command token is in the in-image-binary allowlist.

If a future commit regresses either compose file's ollama healthcheck to `wget`/`curl` (or any other binary added to the forbidden set), `TestOllamaHealthcheck_LiveFiles` will fail with the same shape of error the adversarial sub-cases assert. This converts a `docker compose up -d --wait` exit-124 failure deep inside the test orchestration loop into a fast, named test failure with a clear actionable error message.
