# Design: 020 Security Hardening ŌĆö Docker Binding, Auth Enforcement, Crypto Hygiene

## Design Brief

**Current State:** Smackerel's Docker Compose stack binds postgres, nats, smackerel-ml, and ollama host ports to `0.0.0.0`, exposing them to the LAN. Only `smackerel-core` correctly uses `127.0.0.1`. The ML sidecar (`ml/app/main.py`) has zero HTTP auth middleware. The Web UI route group in `internal/api/router.go` has `webAuthMiddleware` implemented but intentionally not applied. OAuth start/callback endpoints have no rate limiting. `internal/auth/store.go` `decrypt()` silently falls back to plaintext on any failure. NATS auth token is passed as a `--auth` CLI arg visible in `docker ps`.

**Target State:** All host-forwarded ports bind to `127.0.0.1`. ML sidecar validates `X-Auth-Token` / `Authorization: Bearer` on all non-health endpoints. Web UI routes apply `webAuthMiddleware`. OAuth start endpoint is rate-limited. `decrypt()` fails closed when an encryption key is configured. NATS uses a config file mount instead of CLI args. All new config flows through `config/smackerel.yaml` ŌåÆ SST pipeline.

**Patterns to Follow:**
- Port binding pattern from `smackerel-core` in `docker-compose.yml`: `"127.0.0.1:${HOST_PORT}:${CONTAINER_PORT}"`
- Auth middleware pattern from `bearerAuthMiddleware` in [internal/api/router.go](internal/api/router.go) ŌĆö constant-time compare, dev-mode passthrough when token empty
- Config SST pipeline in [scripts/commands/config.sh](scripts/commands/config.sh) ŌĆö `required_value` reads from YAML, writes to generated env
- Fail-loud config validation in [internal/config/config.go](internal/config/config.go)

**Patterns to Avoid:**
- `os.getenv("KEY", "default")` in Python ŌĆö SST requires `os.environ["KEY"]` (fail-loud)
- Shell `${VAR:-default}` fallbacks ŌĆö SST requires `${VAR:?error}`
- Returning raw data on decryption failure (current `store.go` pattern) ŌĆö must return error

**Resolved Decisions:**
- All 6 services use `127.0.0.1` binding in `docker-compose.yml` (including smackerel-core which already does)
- ML sidecar auth uses FastAPI dependency injection, not ASGI middleware
- NATS config file generated by `./smackerel.sh config generate` at `config/generated/nats.conf`
- OAuth rate limiting via `httprate` (Chi ecosystem, already in common usage) ŌĆö 10 req/min per IP on `/auth/{provider}/start`
- `decrypt()` returns `error` when encryption key is present and decryption fails ŌĆö no plaintext fallback
- `config/smackerel.yaml` gains a `host_bind_address` field under `runtime` (default: `127.0.0.1`)

**Open Questions:** None ŌĆö all design decisions are resolved from spec requirements.

## Data Model

No schema changes. No new tables. No migrations.

The only data-adjacent change is that `internal/auth/store.go` `decrypt()` will now return `("", error)` instead of `(rawCiphertext, nil)` on failure. Callers already handle the `error` return ŌĆö the `Get()` method propagates decrypt errors with `fmt.Errorf("decrypt access token for %s: %w", provider, err)`.

**Migration concern:** Existing tokens stored as plaintext (before encryption was enabled) will no longer be silently accepted when an encryption key is present. This is intentional ŌĆö the operator must re-authorize OAuth providers after enabling `auth_token` for the first time. This is documented in R-007's design rationale.

## API / Contracts

### ML Sidecar Auth Contract

All ML sidecar HTTP endpoints (except `/health`) require authentication when `SMACKEREL_AUTH_TOKEN` is non-empty.

**Request header (either):**
- `Authorization: Bearer <token>` (preferred, matches core API convention)
- `X-Auth-Token: <token>` (alternative for service-to-service simplicity)

**Responses:**

| Condition | Status | Body |
|-----------|--------|------|
| Token matches | 2xx | Normal response |
| Token missing or invalid | 401 | `{"detail": "Unauthorized"}` |
| `SMACKEREL_AUTH_TOKEN` empty (dev mode) | 2xx | Normal response (no auth check) |

**Health endpoint (`GET /health`):** Always unauthenticated. Docker healthcheck must work without credentials.

**Implementation ŌĆö FastAPI dependency:**

```python
# ml/app/auth.py

import hmac
import os
from fastapi import Depends, HTTPException, Request


def _get_auth_token() -> str:
    """Read auth token once at import time."""
    return os.environ.get("SMACKEREL_AUTH_TOKEN", "")


_AUTH_TOKEN = _get_auth_token()


async def verify_auth(request: Request) -> None:
    """FastAPI dependency that validates auth token on non-health endpoints.

    When SMACKEREL_AUTH_TOKEN is empty, all requests pass (dev mode).
    When configured, checks Authorization: Bearer or X-Auth-Token header.
    """
    if not _AUTH_TOKEN:
        return

    token = None

    # Check Authorization: Bearer header
    auth_header = request.headers.get("authorization", "")
    if auth_header.lower().startswith("bearer "):
        token = auth_header[7:]

    # Fallback to X-Auth-Token header
    if token is None:
        token = request.headers.get("x-auth-token")

    if token is None or not hmac.compare_digest(token, _AUTH_TOKEN):
        raise HTTPException(status_code=401, detail="Unauthorized")
```

**Applying the dependency to routes in `ml/app/main.py`:**

```python
from .auth import verify_auth

# Health stays unauthenticated
@app.get("/health")
async def health():
    ...

# All other endpoints get the dependency via a router or per-endpoint
from fastapi import APIRouter

authed_router = APIRouter(dependencies=[Depends(verify_auth)])

# Move non-health endpoints to authed_router, then:
app.include_router(authed_router)
```

Since the ML sidecar currently only exposes `/health` via HTTP (all other processing happens via NATS subscriptions), the auth dependency primarily protects against future HTTP endpoints. But the middleware must be in place now so any HTTP endpoint added later is protected by default.

### Core-to-ML Auth

`smackerel-core` already has `SMACKEREL_AUTH_TOKEN` in its environment. When calling the ML sidecar HTTP endpoints (via `ML_SIDECAR_URL`), core must include `Authorization: Bearer <token>` in the request header. The current ML sidecar interaction is NATS-based, so this applies when/if direct HTTP calls are added.

### Web UI Auth Contract

The existing `webAuthMiddleware` in [internal/api/router.go](internal/api/router.go#L149) already implements the correct behavior:
- Empty `AuthToken` ŌåÆ all requests pass (dev mode)
- Non-empty `AuthToken` ŌåÆ checks `Authorization: Bearer` header or `auth_token` cookie
- Failure ŌåÆ `401 Unauthorized`

**Change:** Apply `deps.webAuthMiddleware` to the Web UI route group. The code change is a single line insertion:

```go
// Current (line ~92):
r.Group(func(r chi.Router) {
    r.Get("/", wh.SearchPage)
    ...
})

// After:
r.Group(func(r chi.Router) {
    r.Use(deps.webAuthMiddleware)
    r.Get("/", wh.SearchPage)
    ...
})
```

### OAuth Rate Limiting Contract

**Endpoint:** `GET /auth/{provider}/start`

**Rate limit:** 10 requests per minute per IP address.

**Response when exceeded:** `429 Too Many Requests` with `Retry-After` header.

**Implementation:** Use `github.com/go-chi/httprate` ŌĆö a Chi ecosystem rate limiter that integrates with Chi middleware:

```go
import "github.com/go-chi/httprate"

// In NewRouter, wrap the OAuth start and callback endpoints:
r.Group(func(r chi.Router) {
    r.Use(httprate.LimitByIP(10, 1*time.Minute))
    r.Get("/auth/{provider}/start", oh.StartHandler)
    r.Get("/auth/{provider}/callback", oh.CallbackHandler)
})
// Both endpoints rate-limited per SEC-SWEEP-001 (callback DoS via log flooding)
```

**Dependency addition:** `go get github.com/go-chi/httprate`

## Docker Compose Changes

### Port Binding Format (R-001)

Every `ports:` entry in `docker-compose.yml` must use the `127.0.0.1:` prefix with environment variable substitution. The exact YAML format:

```yaml
# Current (exposed to LAN):
ports:
  - "${POSTGRES_HOST_PORT}:${POSTGRES_CONTAINER_PORT}"

# After (localhost only):
ports:
  - "127.0.0.1:${POSTGRES_HOST_PORT}:${POSTGRES_CONTAINER_PORT}"
```

**All services requiring change:**

| Service | Current | After |
|---------|---------|-------|
| `postgres` | `"${POSTGRES_HOST_PORT}:${POSTGRES_CONTAINER_PORT}"` | `"127.0.0.1:${POSTGRES_HOST_PORT}:${POSTGRES_CONTAINER_PORT}"` |
| `nats` (client) | `"${NATS_CLIENT_HOST_PORT}:${NATS_CLIENT_PORT}"` | `"127.0.0.1:${NATS_CLIENT_HOST_PORT}:${NATS_CLIENT_PORT}"` |
| `nats` (monitor) | `"${NATS_MONITOR_HOST_PORT}:${NATS_MONITOR_PORT}"` | `"127.0.0.1:${NATS_MONITOR_HOST_PORT}:${NATS_MONITOR_PORT}"` |
| `smackerel-core` | `"127.0.0.1:${CORE_HOST_PORT}:${CORE_CONTAINER_PORT}"` | No change (already correct) |
| `smackerel-ml` | `"${ML_HOST_PORT}:${ML_CONTAINER_PORT}"` | `"127.0.0.1:${ML_HOST_PORT}:${ML_CONTAINER_PORT}"` |
| `ollama` | `"${OLLAMA_HOST_PORT}:${OLLAMA_CONTAINER_PORT}"` | `"127.0.0.1:${OLLAMA_HOST_PORT}:${OLLAMA_CONTAINER_PORT}"` |

### NATS Config File Mount (R-006)

Replace CLI `--auth` argument with a config file mount:

```yaml
# Current NATS command:
command: ["--jetstream", "--store_dir", "/data", "--http_port", "${NATS_MONITOR_PORT}", "--auth", "${SMACKEREL_AUTH_TOKEN}"]

# After:
command: ["--config", "/etc/nats/nats.conf"]
volumes:
  - nats-data:/data
  - ./config/generated/nats.conf:/etc/nats/nats.conf:ro
```

**Generated `config/generated/nats.conf` template:**

```
# Auto-generated from config/smackerel.yaml ŌĆö DO NOT EDIT DIRECTLY
# Regenerate: ./smackerel.sh config generate

jetstream {
  store_dir: /data
}

http_port: ${NATS_MONITOR_PORT}

authorization {
  token: ${SMACKEREL_AUTH_TOKEN}
}
```

However, since NATS config files do not support environment variable substitution natively, the config generator must produce a fully resolved file. The generator writes:

```
# Auto-generated from config/smackerel.yaml ŌĆö DO NOT EDIT DIRECTLY
# Regenerate: ./smackerel.sh config generate

jetstream {
  store_dir: /data
}

http_port: 8222

authorization {
  token: "<actual-token-value>"
}
```

Where the port and token are resolved from `config/smackerel.yaml` at generation time.

**Security note:** `config/generated/` is `.gitignore`d. The generated `nats.conf` contains the resolved auth token and must never be committed.

2. **Write to env file:**

## Decryption Fail-Closed Logic (R-007)

### Current Behavior (`internal/auth/store.go`)

The `decrypt()` method has 3 fallback paths that silently return the encoded input as plaintext:

1. **Not base64:** `slog.Warn("token not base64-encoded, treating as plaintext")` ŌåÆ returns `encoded`
2. **Too short:** `slog.Warn("token too short for encrypted data, treating as plaintext")` ŌåÆ returns `encoded`
3. **GCM Open failure:** `slog.Warn("token decryption failed, treating as plaintext")` ŌåÆ returns `encoded`

### New Behavior

When `s.encKey` is non-nil (encryption key derived from non-empty `auth_token`):

1. **Not base64:** Return `("", fmt.Errorf("token is not valid base64: %w", err))`
2. **Too short:** Return `("", fmt.Errorf("encrypted token data too short (got %d bytes, need at least %d)", len(data), nonceSize))`
3. **GCM Open failure:** Return `("", fmt.Errorf("token decryption failed: %w", err))`

When `s.encKey` is nil (no encryption key ŌĆö dev mode):
- No change. Returns `(encoded, nil)` ŌĆö plaintext passthrough.

### Implementation

```go
func (s *TokenStore) decrypt(encoded string) (string, error) {
    if len(s.encKey) == 0 || encoded == "" {
        return encoded, nil
    }

    data, err := base64.StdEncoding.DecodeString(encoded)
    if err != nil {
        return "", fmt.Errorf("token is not valid base64: %w", err)
    }

    block, err := aes.NewCipher(s.encKey)
    if err != nil {
        return "", fmt.Errorf("create cipher: %w", err)
    }

    gcm, err := cipher.NewGCM(block)
    if err != nil {
        return "", fmt.Errorf("create GCM: %w", err)
    }

    nonceSize := gcm.NonceSize()
    if len(data) < nonceSize {
        return "", fmt.Errorf("encrypted token data too short (got %d bytes, need at least %d)", len(data), nonceSize)
    }

    nonce, ciphertext := data[:nonceSize], data[nonceSize:]
    plaintext, err := gcm.Open(nil, nonce, ciphertext, nil)
    if err != nil {
        return "", fmt.Errorf("token decryption failed: %w", err)
    }

    return string(plaintext), nil
}
```

### Migration Impact

Tokens stored as plaintext before `auth_token` was configured will fail to decrypt once `auth_token` is set. This is the correct behavior ŌĆö the operator should re-authorize OAuth providers after enabling encryption. The error message is clear and actionable. The `Get()` method already propagates this error to callers with provider context.

### Constant-Time Comparison

All token comparisons use constant-time functions:
- Go: `crypto/subtle.ConstantTimeCompare` (existing in `router.go`)
- Python: `hmac.compare_digest` (new in `ml/app/auth.py`)

### No Secrets in Logs

Auth failure log messages include request path and source IP but never include token values (neither the expected token nor the submitted token).

### NATS Token Visibility Fix

Moving from `--auth <token>` CLI arg to `nats.conf` file mount removes the token from `docker ps` output. The config file is mounted read-only (`:ro`) and lives in the gitignored `config/generated/` directory.

## Risks & Open Questions

**Resolved risks (no open questions):**

1. **Pre-existing plaintext tokens:** After enabling `auth_token`, stored OAuth tokens encrypted before key derivation will fail to decrypt. This is intentional fail-closed behavior. Operators re-authorize OAuth providers ŌĆö documented in R-007.

2. **NATS config file permissions:** Docker volume mount with `:ro` ensures the container cannot modify the file. Host file permissions are managed by the config generator (standard `umask`). For hardened deployments, operators can set `chmod 600` on `config/generated/nats.conf`.

3. **Rate limiter state:** `httprate` uses in-memory state by default. Rate limit state is lost on restart, which is acceptable for this use case (OAuth start endpoint, single-instance deployment).

4. **ML sidecar HTTP surface:** Currently only `/health` is exposed via HTTP; all processing happens via NATS. The auth middleware is still required because (a) it protects any future HTTP endpoints by default and (b) the port is still forwarded and reachable on localhost.

## Alternatives Considered

### Port Binding: `HOST_BIND_ADDRESS` Variable in Compose

Considered using `"${HOST_BIND_ADDRESS}:${HOST_PORT}:${CONTAINER_PORT}"` in `docker-compose.yml` with the env var set by config generation. Rejected because Docker Compose environment variable substitution in port strings has inconsistent behavior across Compose versions, and a literal `127.0.0.1` is simpler, clearer, and guaranteed correct. The SST field exists for audit and documentation.

### ML Sidecar Auth: ASGI Middleware vs FastAPI Dependency

ASGI middleware would intercept all requests including `/health`. A FastAPI dependency can be selectively applied per-router, keeping `/health` unauthenticated for Docker healthcheck. Dependency injection is the idiomatic FastAPI pattern and matches the requirement exactly.

### NATS Auth: Docker Secrets vs Config File Mount

Docker Secrets require Swarm mode or Compose v3 `secrets:` support, which adds complexity. A simple read-only bind mount of a generated config file achieves the same goal (token not in CLI args) with minimal change to the existing Compose setup.

### Decrypt Fallback: Warning + Return vs Error

The current "warn and return plaintext" approach was designed for migration from unencrypted to encrypted storage. The spec explicitly requires fail-closed (R-007). Returning an error when an encryption key is present is the correct security posture ŌĆö silent degradation masks data corruption and potential attacks.
