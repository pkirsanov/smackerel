## Root Cause Analysis

### Defect A & B — `err_code=10100 "filtered consumer not unique on workqueue stream"`

The Python ML sidecar at startup creates permanent durable JetStream consumers
via `pull_subscribe("artifacts.process", durable=...)` and
`pull_subscribe("domain.extract", durable=...)` — see
[ml/app/nats_client.py](../../../../ml/app/nats_client.py) (`SUBSCRIBE_SUBJECTS`,
line 25 onward; subscription loop around line 132).

The ARTIFACTS and DOMAIN streams use `Retention: WorkQueuePolicy`. NATS
JetStream forbids two consumers on a workqueue stream from holding
overlapping `FilterSubject` values — exact-string filters with the same
value overlap. The integration tests originally created a second consumer
with the **exact same** `FilterSubject` (`"artifacts.process"` /
`"domain.extract"`), which NATS rejected immediately with `code=400
err_code=10100 description="filtered consumer not unique on workqueue
stream"`. This is a deterministic test-isolation defect, not a defect in
production wiring.

**Fix:** the test now creates and publishes to a per-invocation subject
(`artifacts.process.<test-id>` / `domain.extract.<test-id>`) which still
matches the streams' `"artifacts.>"` / `"domain.>"` subject patterns but
does NOT overlap with the ML sidecar's exact-subject filter. This is
design.md option **B (non-overlapping filter)** and preserves real
production wiring untouched. No consumer pre-deletion shortcut was used.

### Defect C — `expected 0 messages after MaxDeliver exhaustion, got 1`

The chaos consumer originally used `FilterSubject: "deadletter.>"` (a
wildcard) against the DEADLETTER stream which is `Retention: LimitsPolicy`
with a 30-day `MaxAge`. Stale `deadletter.*` messages from prior failed
test runs (or other tests in the integration package such as
`TestNATS_ConsumerReplay_NakRedeliver`) accumulate in the stream because
LimitsPolicy does not consume-and-delete on ack. With the wildcard filter
and the default `DeliverAllPolicy`, the test's 3 `Nak()` attempts were
being absorbed by an **older** poison message; the post-exhaustion
`cons.Fetch(1, ...)` then pulled the still-pending **fresh** poison message
(or another stale one), which the test attributed to a MaxDeliver semantics
break.

MaxDeliver was, in fact, working correctly all along. The defect was
test-isolation: a wildcard filter on a long-retained stream cross-contaminates
between test invocations.

**Fix:** scope the chaos consumer's `FilterSubject` to the unique per-test
subject (`deadletter.chaos-maxdeliver-<test-id>`) so only this test's
freshly-published poison message can match. After 3 NAKs MaxDeliver=3 is
reached for the only candidate message; the post-exhaustion fetch returns
zero, exactly as the contract requires. The `extraReceived == 0` assertion
is preserved; no `<= 1` softening, no arbitrary sleep introduced (existing
`time.Sleep(2 * time.Second)` past `AckWait` is unchanged). This is
design.md candidate option B/D semantics (per-subject isolation rather than
advisory-driven DLQ wiring, which would have required production changes
outside the chosen change boundary).
