# Bug: BUG-029-003 — Dev `docker-compose.yml` violates Gate G028 NO-DEFAULTS via 14 `${VAR:-default}` substitutions on build metadata, env-file path, image refs, and volume mounts

## Classification

- **Type:** DevOps defect — dev-compose Gate G028 violation sweep (operational hardening)
- **Severity:** P3 — LOW (the dev compose file `docker-compose.yml` is used only by `./smackerel.sh up` / `./smackerel.sh test` workflows; the production / home-lab compose file `deploy/compose.deploy.yml` is unaffected and was already converted to fail-loud forms by spec 042 and BUG-042-001..005). The risk of the pre-fix state is silent SST drift: a developer who manually invokes `docker compose -f docker-compose.yml up` without first running `./smackerel.sh config generate` would see the dev image silently use `dev / unknown / unknown` for VERSION / COMMIT / BUILD_TIME and silently fall back to `config/generated/dev.env` for the env-file path, masking real misconfiguration.
- **Parent Spec:** 029 — DevOps Pipeline & Image Governance (owns `./smackerel.sh up` workflow + dev `docker-compose.yml` + `./smackerel.sh config generate` SST chain)
- **Workflow Mode:** test-to-doc
- **Status:** Fixed
- **Discovered By:** 2026-05-14 home-lab readiness re-scan (finding HL-RESCAN-012)

## Problem Statement

The dev compose file `docker-compose.yml` is the entrypoint Compose file that `./smackerel.sh up` invokes (via `scripts/lib/runtime.sh::smackerel_compose`) to bring the local development stack online (Postgres, NATS, smackerel-core, smackerel-ml, optional Ollama, optional Prometheus). The Smackerel runtime contract per `.github/copilot-instructions.md` is:

> **SST Zero-Defaults Enforcement (NON-NEGOTIABLE)** — ALL configuration values MUST originate from `config/smackerel.yaml`. Zero hardcoded ports, URLs, hostnames, or fallback defaults anywhere in the codebase.
>
> | Language | FORBIDDEN | REQUIRED |
> |----------|-----------|----------|
> | **Shell** | `${VAR:-default}` with fallback | `${VAR:?error message}` fail-loud |

Docker Compose's `${VAR}` substitution is shell-syntax-compatible: the `:-` form is the silent-fallback form that violates Gate G028, and the `:?` form is the required fail-loud form. The pre-fix `docker-compose.yml` had **14 occurrences of the forbidden `${VAR:-default}` form**, classified into four categories:

1. **Build metadata (6 occurrences):** `${SMACKEREL_VERSION:-dev}`, `${SMACKEREL_COMMIT:-unknown}`, `${SMACKEREL_BUILD_TIME:-unknown}` × 2 services (smackerel-core, smackerel-ml). These are the labels and ldflags that bake VERSION / COMMIT / BUILD_TIME into the binary; the silent fallback means a developer who forgets to set these gets `dev / unknown / unknown` baked into the image with no warning.
2. **Env-file path (2 occurrences):** `${SMACKEREL_ENV_FILE:-config/generated/dev.env}` × 2 services. These are the `env_file:` references; the silent fallback means a developer who runs `docker compose up` directly (instead of `./smackerel.sh up`) silently picks up `config/generated/dev.env` even when a different env file was intended.
3. **Image refs (2 occurrences):** `${SMACKEREL_CORE_IMAGE:-}` and `${SMACKEREL_ML_IMAGE:-}`. Empty default is the build-from-source toggle: if the operator sets a registry digest in `SMACKEREL_CORE_IMAGE`, Compose pulls that image; otherwise (empty), Compose builds from the local Dockerfile context. The `:-` form here is wrong because it conflates "explicitly empty (build from source)" with "unset (operator forgot)". The correct fail-loud form is `${SMACKEREL_CORE_IMAGE?...}` (no colon — fails on unset only, allows explicitly empty).
4. **Volume mounts (4 occurrences):** `${BOOKMARKS_IMPORT_DIR:-./data/bookmarks-import}`, `${MAPS_IMPORT_DIR:-./data/maps-import}`, `${BROWSER_HISTORY_PATH:-./data/browser-history}`, `${TWITTER_ARCHIVE_DIR:-./data/twitter-archive}`. These are the connector-fixture mount points. The empty value pattern is **load-bearing**: when the env var is empty, the connector treats it as "no import configured" via the `${X:+/data/...}` env override pattern in the connector code itself. Converting the volume mount to fail-loud requires the connector code to be refactored to handle the SST-emitted "explicit empty" signal differently from "unset". This is OUT OF SCOPE for HL-RESCAN-012 (P3) and is filed as a follow-up.

The runtime behavior of the smackerel-core / smackerel-ml services was correct in practice — `./smackerel.sh up` invokes `scripts/lib/runtime.sh::smackerel_compose` which always sets `--env-file config/generated/dev.env`, so the silent fallback never fires in the sanctioned developer workflow. But the contract surface had a gap: a developer who manually invokes `docker compose -f docker-compose.yml up` (or any tool that does not first call `./smackerel.sh config generate`) would silently get the fallback values, masking real misconfiguration. The fix wires build metadata + env-file path + image refs through the SST as fail-loud-substituted vars, and adds a static-file lint test that asserts the dev compose file is contract-compliant (with an explicit allowlist for the four volume-mount load-bearing exceptions).

## Acceptance Criteria

- AC-1: `scripts/commands/config.sh` emits `SMACKEREL_VERSION`, `SMACKEREL_COMMIT`, `SMACKEREL_BUILD_TIME`, `SMACKEREL_CORE_IMAGE`, `SMACKEREL_ML_IMAGE` into the generated `config/generated/<env>.env` file. The values default to safe placeholders (`dev` / `unknown` / `unknown` / empty / empty) at SST resolution time but can be overridden by setting the corresponding shell env var BEFORE invoking `./smackerel.sh config generate` (so CI / release pipelines that already export `SMACKEREL_VERSION` and `SMACKEREL_COMMIT` continue to flow through the SST).
- AC-2: `docker-compose.yml` uses the fail-loud `${X:?error message}` form for `SMACKEREL_VERSION`, `SMACKEREL_COMMIT`, `SMACKEREL_BUILD_TIME`, and `SMACKEREL_ENV_FILE` on both `smackerel-core` and `smackerel-ml` services. Each error message names the regression target (Gate G028 / HL-RESCAN-012) and the fix path (`run ./smackerel.sh config generate` or `./smackerel.sh up`).
- AC-3: `docker-compose.yml` uses the fail-on-unset-but-allow-empty `${X?error message}` form (no colon — empty values are accepted) for `SMACKEREL_CORE_IMAGE` and `SMACKEREL_ML_IMAGE`. Each error message documents the build-from-source toggle: explicitly-empty value means "build from local Dockerfile"; unset (var entirely missing) is a real misconfiguration that Compose aborts on.
- AC-4: A new persistent in-tree adversarial test `internal/deploy/dev_compose_default_fallback_test.go` exists in the same `package deploy` as the existing `compose_contract_test.go`. The test parses the live `docker-compose.yml` line-by-line, regex-matches every `${VAR:-default}` occurrence, filters via an explicit per-var allowlist, and asserts zero unauthorized matches.
- AC-5: The new test's allowlist names exactly the four load-bearing volume-mount vars (`BOOKMARKS_IMPORT_DIR`, `MAPS_IMPORT_DIR`, `BROWSER_HISTORY_PATH`, `TWITTER_ARCHIVE_DIR`) with a one-line justification each ("connector-fixture mount; empty value = no import (consumed via `${X:+/data/...}` env override)"). Each justification documents the deferral path (follow-up bug to refactor the connector code to consume an SST-emitted "explicit empty" signal).
- AC-6: The new test has at least three adversarial sub-cases: (a) regression of any build-metadata / env-file / image-ref var to `:-` form is caught; (b) a novel non-allowlisted var added in `:-` form is caught; (c) per-var allowlist gating works (allowlist is keyed by var name, not line number — adding the same var in a NEW location is allowed; adding a different var in `:-` form on the SAME line as an allowlisted one is NOT allowed). One additional adversarial sub-case proves comment-line skipping: a line that is a Gate G028 comment containing the literal text `${VAR:-default}` does NOT trigger a false positive.
- AC-7: HL-RESCAN-012 attribution is present in either the test docstring or the failure-case error message, so a future regression points back to this bug.
- AC-8: `docker compose --env-file config/generated/dev.env -f docker-compose.yml config -q` exits 0 (the regenerated env file satisfies the new fail-loud substitution forms). Same check for `config/generated/test.env`. RED proof: `docker compose --env-file /dev/null -f docker-compose.yml config -q` exits non-zero with a Compose error message that names at least one of the new fail-loud-substituted vars (proving the fail-loud forms are active and Compose aborts on unset).
- AC-9: `TestComposeContract_LiveFile` and the eight pre-existing adversarial tests in `internal/deploy/compose_contract_test.go` (which lock the prod compose file `deploy/compose.deploy.yml`) continue to PASS GREEN unchanged — the new dev-compose test is purely additive against a different file (`docker-compose.yml`) and does not over-reach into the prod-compose contract surface.

## Out of Scope

- Converting the four volume-mount `${X:-./data/...}` defaults to fail-loud forms. The empty value is load-bearing for the connector code's "no import configured" signal. Converting them requires (a) emitting `BOOKMARKS_IMPORT_DIR` / `MAPS_IMPORT_DIR` / `BROWSER_HISTORY_PATH` / `TWITTER_ARCHIVE_DIR` from the SST as truly empty (already done) and (b) the connector code recognising the difference between "explicit empty SST value (no import)" and "unset env var (real misconfiguration)" — which is a connector-side refactor outside the dev-compose contract surface. Filed as a follow-up bug.
- Editing `deploy/compose.deploy.yml` (the prod / home-lab compose file). Already locked by spec 042 + BUG-042-001..005 with `internal/deploy/compose_contract_test.go`.
- Editing `specs/029-devops-pipeline/spec.md`, `specs/029-devops-pipeline/design.md`, `specs/029-devops-pipeline/scopes.md`, `specs/029-devops-pipeline/state.json`, `specs/029-devops-pipeline/uservalidation.md`, or `specs/029-devops-pipeline/report.md` — foreign-owned parent-spec content; outside `bubbles.devops` mode edit scope.
- Editing `.github/workflows/ci.yml` to export `SMACKEREL_VERSION` / `SMACKEREL_COMMIT` differently. The existing workflow already exports both via `git describe` / `${{ github.sha }}` before invoking `./smackerel.sh build`; the SST generator change in this fix preserves that flow (the resolution order is "shell env wins; placeholder fallback is only used when neither shell env nor `config/smackerel.yaml` provides a value").
- Cosign-signing or attesting the dev compose file itself (the dev compose file is not a release artifact — only the prod compose file under `deploy/` is part of the build-once-deploy-many surface).
